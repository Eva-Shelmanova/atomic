# Ответы на вопросы оппонентов по проекту atomic

---

## Архитектура и дизайн (1–20)

**1. Почему выбран паттерн Chain of Responsibility, а не LangGraph или Semantic Kernel?**  
Chain of Responsibility — минималистичный и понятный паттерн для небольшого числа агентов (3–4). LangGraph и Semantic Kernel дают больше гибкости, но требуют больше зависимостей и усложняют архитектуру. Для учебного/демо-проекта CoR достаточен; при необходимости можно мигрировать на LangGraph.

**2. Почему RouterAgent использует LLM для классификации, а не rule-based?**  
LLM лучше справляется с размытыми формулировками и синонимами. Rule-based подходы требуют регулярного обновления правил и хуже масштабируются на новые типы запросов. LLM даёт гибкость при небольшом overhead (один вызов на запрос).

**3. Как обоснован порядок агентов: Retriever → Command → Analyst?**  
Retriever — для быстрого доступа к знаниям; Command — для явных действий. Analyst — fallback для всех остальных случаев. Порядок отражает приоритет: сначала факты и действия, потом рассуждения.

**4. Почему AnalystAgent всегда can_handle=True?**  
Analyst — универсальный fallback. Если ни Retriever, ни Command не обработали запрос, Analyst отвечает. Это гарантирует, что каждый запрос получит ответ.

**5. Как бы вы масштабировали систему при росте числа агентов?**  
Добавить реестр агентов с приоритетами; вынести конфигурацию в YAML/JSON; рассмотреть LangGraph для сложных графов; добавить кэширование результатов роутера.

**6. Почему нет явного State Machine для workflow?**  
Текущий workflow линейный: один запрос → один ответ. State Machine имеет смысл при multi-step или условных переходах. Для MVP достаточно линейной цепочки.

**7. Как бы вы добавили нового агента (например, для работы с API)?**  
Создать класс, наследующий BaseAgent; реализовать can_handle() и process(); добавить в _get_chain(); расширить Intent и ROUTER_PROMPT; при необходимости добавить новый инструмент (например, API client).

**8. Почему workflow синхронный, а не асинхронный?**  
Оригинальный Ollama client и psycopg2 синхронны. Асинхронность потребует async-версий (httpx, asyncpg) и усложнит код. Для локального/демо-приложения синхронность приемлема.

**9. Как обеспечивается идемпотентность при повторных запросах?**  
Чистый запрос без побочных эффектов (чтение, поиск) идемпотентен. CommandAgent выполняет код (с побочными эффектами). Для идемпотентности можно добавить idempotency key и кэш результатов.

**10. Почему ChatMemory и VectorMemory разделены?**  
ChatMemory — краткосрочная история диалога (до 20 сообщений). VectorMemory — долгосрочная база знаний. Разные задачи: контекст диалога vs. семантический поиск. Объединение усложнило бы логику и хранение.

**11. Как бы вы реализовали multi-turn planning?**  
Планировщик: LLM разбивает задачу на шаги; каждый шаг — отдельный вызов workflow; результаты сохраняются в контексте; цикл до завершения или ошибки. Можно добавить PlannerAgent.

**12. Почему роутер вызывается до цепочки агентов?**  
Роутер определяет intent один раз и передаёт его в контекст. Агенты используют intent в can_handle(). Вызов роутера внутри цепочки привёл бы к дублированию LLM-вызовов.

**13. Как бы вы добавили параллельное выполнение нескольких агентов?**  
Если агенты независимы — asyncio.gather или ThreadPoolExecutor. В текущей цепочке один агент обрабатывает запрос, параллелизм не нужен. Для multi-agent можно использовать LangGraph с параллельными ветками.

**14. Почему нет явного слоя валидации входных данных?**  
Валидация в API: Pydantic ChatRequest. Для production стоит добавить валидацию длины, санитизацию, фильтрацию запрещённых символов. В README это указано как возможное расширение.

**15. Как бы вы реализовали отмену (cancellation) долгого запроса?**  
В async: asyncio.CancelledError или Task.cancel(). В синхронном: threading.Event или timeout. Для Ollama — проверка флага в цикле между вызовами; для subprocess — process.terminate().

**16. Почему нет явного error-handling на уровне workflow?**  
Ошибки LLM возвращают строку вида [Ошибка LLM: ...]. При ошибке PostgreSQL — исключение. Для production стоит добавить try/except, логирование, fallback-ответы и retry.

**17. Как бы вы добавили retry при сбоях LLM?**  
Обёртка: tenacity или retrying; retry на ConnectionError, Timeout; exponential backoff; ограничение числа попыток. Можно добавить в LLMClient.chat().

**18. Почему не используется event-driven подход?**  
Event-driven (очереди, Kafka) подходит для высокой нагрузки и асинхронной обработки. Текущий проект — синхронный request-response. Для production с очередями event-driven можно рассмотреть.

**19. Как бы вы реализовали A/B-тестирование разных агентов?**  
Конфигурация: несколько вариантов агентов; случайный выбор по user_id или session_id; логирование выбранного варианта и результата; анализ в Langfuse или отдельной системе.

**20. Почему нет явного контракта (schema) между агентами?**  
AgentResponse — dataclass с content, agent_type, handled, metadata. Явный Pydantic schema улучшит валидацию и документацию. Для расширения можно ввести AgentResponseSchema.

## RAG и векторный поиск (21–35)

**21. Почему документы не разбиваются на чанки?**  
Для MVP и небольших датасетов (dataset/*.txt) целый документ — один чанк. Это упрощает загрузку и подходит для коротких инструкций. Chunking — в report.txt как направление развития.

**22. Как вы оцениваете качество retrieval?**  
DeepEval тесты проверяют релевантность ответов. Для recall@k и MRR нужны размеченные данные и метрики. Можно добавить: golden dataset, eval-скрипт, сравнение с baseline.

**23. Почему top_k=5, а не настраиваемый параметр?**  
5 — разумный компромисс для контекста и стоимости. Для production: вынести в config или env; добавить в API; подбирать по метрикам.

**24. Почему выбран косинус, а не L2 или inner product?**  
nomic-embed-text обычно нормализует векторы; косинус эквивалентен inner product для нормализованных векторов. pgvector поддерживает vector_cosine_ops. L2 — для ненормализованных векторов.

**25. Почему embedding dimension 768?**  
nomic-embed-text выдаёт 768 измерений. Размерность задаётся моделью; ATOMIC_EMBEDDING_DIM переопределяет при необходимости.

**26. Как бы вы реализовали semantic chunking?**  
Разбиение по смыслу: LLM или sentence-transformers; границы по абзацам/предложениям; overlap; сохранение иерархии (parent_id). Библиотеки: langchain, semantic-kernel.

**27. Почему нет гибридного поиска (BM25 + vector)?**  
В report.txt гибридный поиск описан как перспектива. Для MVP достаточно векторного. Добавление: pgvector + full-text search; взвешенный score; RRF.

**28. Как обрабатывается ситуация, когда в базе нет релевантных документов?**  
RetrieverAgent: retrieved = "Нет релевантных документов в базе знаний."; промпт просит честно сказать об этом. LLM возвращает сообщение об отсутствии релевантной информации.

**29. Почему нет re-ranking после первичного retrieval?**  
Cross-encoder re-ranking повышает качество, но увеличивает latency и стоимость. Для небольшого top_k=5 можно обойтись без него. Для production — cross-encoder или LLM-as-reranker.

**30. Как бы вы обновили индекс при изменении документов?**  
Удаление по metadata (source); перезагрузка через load_dataset.py; или soft delete + upsert. Для инкрементальных обновлений — версионирование по source или hash.

**31. Почему используется HNSW, а не IVF?**  
HNSW — хороший баланс для read-heavy и средних объёмов. pgvector поддерживает HNSW. IVF — для очень больших индексов. В текущем масштабе HNSW достаточен.

**32. Как вы оцениваете дрейф эмбеддингов со временем?**  
Модуль не реализован. Для production: периодическое пересчитывание эмбеддингов; сравнение с baseline; мониторинг метрик retrieval; A/B тесты.

**33. Почему нет metadata filtering при поиске?**  
В PgVectorStore metadata хранится, но не используется в WHERE. Для production: фильтр по source, дате, типу; pgvector поддерживает фильтрацию в запросе.

**34. Как бы вы добавили multi-lingual retrieval?**  
Модель эмбеддингов с поддержкой нескольких языков (например, multilingual-e5); или отдельные индексы по языкам. nomic-embed-text неплохо работает с русским.

**35. Почему нет query expansion перед поиском?**  
Query expansion (синонимы, переформулировки) повышает recall. Для MVP не реализовано. Добавление: LLM для расширения запроса; или несколько вариантов запроса и объединение результатов.

## Безопасность и CodeExecutor (36–45)

**36. Какие риски несёт выполнение произвольного Python-кода через subprocess?**  
Доступ к файловой системе, сеть, бесконечные циклы, потребление памяти. В README указано: «для production нужен полноценный sandbox».

**37. Почему CodeExecutor не использует Docker sandbox?**  
Docker — для production. В демо/учебном проекте subprocess — быстрее и проще. Для production: Docker, E2B, Lambda, изолированные сервисы.

**38. Как ограничить доступ к файловой системе при выполнении кода?**  
В subprocess — ограничено. В Docker: read-only root, tmpfs; ограничение capabilities. В E2B: изолированная VM. Можно добавить whitelist/blacklist путей.

**39. Есть ли защита от prompt injection в CommandAgent?**  
Косвенная: промпт просит «только код». Явной защиты нет. Для production: санитизация ввода; разделение инструкций и данных; ограничение формата ответа.

**40. Как предотвратить выполнение вредоносного кода (os.system, subprocess и т.п.)?**  
В subprocess — нельзя. В Docker: ограничение capabilities, network. В E2B: sandbox. Можно добавить статический анализ (ast) — запрет import os, subprocess и т.п.

**41. Почему timeout 10 секунд — достаточен ли он?**  
Для простых вычислений — да. Для тяжёлых вычислений — нет. Вынести в config; для production — настраиваемый параметр, возможно 30–60 с.

**42. Как изолировать выполнение кода между пользователями?**  
Сейчас один глобальный CodeExecutor. Для production: отдельный sandbox на user/session; ограничение ресурсов; очистка между запусками.

**43. Есть ли rate limiting на выполнение кода?**  
Нет. Для production: per-user лимит; очередь; ограничение частоты вызовов CommandAgent.

**44. Как защитить от DoS через тяжёлые вычисления?**  
Timeout (10 с); rate limiting; очередь с ограничением. В Docker — ограничение CPU/memory. Для production — отдельный сервис с ограничениями.

**45. Почему код выполняется в tempfile, а не в полностью изолированной среде?**  
Tempfile — минимальная реализация для демо. subprocess.run уже даёт отдельный процесс. Полная изоляция — Docker/E2B или аналоги.

## Langfuse и observability (46–55)

**46. Почему Langfuse опционален, а не обязателен?**  
Langfuse требует ключи и (опционально) self-hosted. Для локального запуска без трейсинга не нужен. Опциональность упрощает onboarding.

**47. Как session_id влияет на группировку трасс?**  
propagate_attributes(session_id=...) помечает все дочерние spans. В Langfuse Sessions трассы с одним session_id группируются для replay сессии.

**48. Почему session_id ограничен 200 символами?**  
Ограничение Langfuse API: «sessionId can be any US-ASCII character string less than 200 characters». В коде: session_id[:200].

**49. Что логируется при ошибке LLM?**  
LLMClient возвращает строку [Ошибка LLM: ...]. observe_llm вызывается с этим output. Langfuse получает trace. Явного логирования ошибок в файл нет — можно добавить.

**50. Как бы вы добавили метрики latency по агентам?**  
Замер времени в process() каждого агента; отправка в Langfuse metadata или через OpenTelemetry; дашборд в Langfuse или Grafana.

**51. Почему нет явного trace_id для distributed tracing?**  
Langfuse генерирует trace_id. Для distributed tracing можно явно передавать trace_id в заголовках и использовать W3C Trace Context.

**52. Как Langfuse интегрируется с API и CLI?**  
CLI: session_id генерируется один раз на сессию. API: session_id из body или X-Session-ID. trace_ask вызывается в workflow.ask() с session_id.

**53. Почему используется propagate_attributes, а не ручная передача session_id?**  
propagate_attributes распространяет session_id на все дочерние spans (LLM, embedding) через OpenTelemetry context. Ручная передача потребовала бы изменения всех вызовов.

**54. Как бы вы добавили алерты при падении качества?**  
Метрики в Langfuse; интеграция с Slack/PagerDuty; пороги по latency, error rate, score; периодические eval-запуски.

**55. Почему Langfuse, а не OpenTelemetry напрямую?**  
Langfuse — готовый продукт для LLM-трейсинга; OTel — низкоуровневый протокол. Langfuse использует OTel под капотом. Для LLM-специфичных метрик Langfuse удобнее.

## API и интеграция (56–65)

**56. Почему API синхронный, а не асинхронный?**  
Workflow синхронный; FastAPI поддерживает sync и async. Для async нужен async-стек (httpx, asyncpg). Для текущей нагрузки sync приемлем.

**57. Почему один глобальный workflow для всех пользователей?**  
Упрощение для MVP. ChatMemory — in-memory, shared. Для production: workflow на user/session; Redis или БД для ChatMemory.

**58. Как бы вы реализовали per-user session и ChatMemory?**  
Ключ по user_id/session_id; хранилище (dict, Redis); создание workflow при первом запросе или dependency injection. В lifespan — фабрика workflow.

**59. Почему нет rate limiting на API?**  
Для демо не требуется. Для production: slowapi, fastapi-limiter; лимиты по IP/user; защита от DoS.

**60. Почему нет аутентификации?**  
Демо-проект. Для production: JWT, OAuth, API keys; middleware для проверки; RBAC.

**61. Как бы вы добавили streaming ответов?**  
StreamingResponse; генератор, yield chunks; LLM streaming (Ollama stream=True); Server-Sent Events или WebSocket.

**62. Почему session_id можно передавать и в body, и в заголовке?**  
Удобство: body — для JSON-клиентов; заголовок — для прокси, CDN, без изменения тела. Приоритет: body > заголовок.

**63. Почему нет версионирования API?**  
MVP. Для production: /v1/chat; или заголовок Accept-Version; обратная совместимость.

**64. Как бы вы добавили WebSocket для real-time чата?**  
FastAPI WebSocketEndpoint; цикл приёма сообщений; вызов workflow.ask(); отправка ответа. Для streaming — отправка по частям.

**65. Почему нет CORS-конфигурации в документации?**  
CORS нужен для браузерных клиентов. Для production: CORSMiddleware; настройка origins. В демо — не указано.

## База данных и Docker (66–75)

**66. Почему PostgreSQL, а не Qdrant, Weaviate или Pinecone?**  
PostgreSQL + pgvector: одна БД для приложения и векторов; знакомый стек; self-hosted. Qdrant/Weaviate — специализированные векторы; Pinecone — managed. Для учебного проекта pgvector удобен.

**67. Почему два docker-compose (atomic и Langfuse)?**  
Разные цели: atomic — PostgreSQL для RAG; Langfuse — трейсинг. Разделение: можно запускать независимо; Langfuse опционален.

**68. Как решается конфликт портов PostgreSQL на 5432?**  
Langfuse Postgres проброшен на 5433; atomic Postgres — на 5432. Конфликта нет.

**69. Почему Langfuse Postgres на порту 5433?**  
Чтобы не конфликтовать с atomic-postgres на 5432. Внутри Docker сети Langfuse использует postgres:5432.

**70. Как бы вы масштабировали PostgreSQL при росте нагрузки?**  
Connection pooling (pgbouncer); read replicas; партиционирование; при необходимости — выделенная векторная БД (Qdrant, Weaviate).

**71. Почему нет миграций для схемы atomic_documents?**  
init_schema() создаёт таблицу при первом подключении. CREATE TABLE IF NOT EXISTS. Для production: Alembic или Flyway; версионирование изменений.

**72. Как бы вы реализовали backup/restore?**  
pg_dump/pg_restore; cron для периодического backup; хранение в S3. Для volume — backup docker volume.

**73. Почему volume atomic_pgdata, а не bind mount?**  
Named volume — Docker управляет; переносимость; производительность на Linux. Bind mount — для доступа к данным с хоста.

**74. Как бы вы добавили read replicas?**  
PostgreSQL streaming replication; отдельный DSN для чтения; в PgVectorStore — два connection: read/write. Для поиска — read replica.

**75. Почему нет healthcheck для Ollama в docker-compose?**  
Ollama — внешний сервис, не в docker-compose atomic. Для production: отдельный compose с Ollama; healthcheck; проверка перед стартом.

## Промпты и LLM (76–85)

**76. Почему RouterAgent использует temperature=0.1, а Analyst — 0.5?**  
Router — нужна детерминированность (classification). Analyst — креативность и разнообразие. Retriever/Command — 0.2–0.3 (баланс).

**77. Как бы вы защитили промпты от prompt injection?**  
Разделение инструкций и данных; экранирование; валидация; output parsing (только ожидаемый формат). Для production — input sanitization, мониторинг.

**78. Почему в RETRIEVER_PROMPT нет инструкции по языку ответа?**  
Промпт Analyst: «Отвечай на том же языке, что и вопрос». Для Retriever можно добавить: «Отвечай на языке вопроса».

**79. Как бы вы добавили few-shot примеры в промпты?**  
Добавить в ROUTER_PROMPT, RETRIEVER_PROMPT и т.д. примеры input/output; хранить в config или отдельном файле; подставлять в шаблон.

**80. Почему max_tokens различаются для разных агентов?**  
Router — нужен короткий ответ (1 слово) → 20. Retriever — 1024. Analyst — 2048 (развёрнутые ответы). Command — 1024 (код).

**81. Как бы вы реализовали prompt versioning?**  
Хранить промпты в БД или YAML; версия в metadata; A/B тесты; логирование в Langfuse.

**82. Почему нет system prompt для агентов?**  
Инструкции в user message. System prompt — для роли/персоны. Можно добавить: `messages=[{"role": "system", "content": "..."}, {"role": "user", ...}]`.

**83. Как бы вы оценивали качество ответов через LLM-as-a-Judge?**  
DeepEval AnswerRelevancyMetric (маркер slow). Альтернатива: отдельный LLM; промпт: «Оцени релевантность 1–5»; агрегация по датасету.

**84. Почему используются разные модели для Router и Retriever?**  
Router — быстрая, дешёвая модель (qwen). Retriever — более мощная (deepseek-r1) для сложных запросов. Разделение по cost/quality.

**85. Как бы вы добавили fallback на другую модель при ошибке?**  
try/except в LLMClient; при ошибке — вызов с другой моделью из списка; fallback_models в config.

## Тестирование (86–92)

**86. Почему тесты помечены как slow и fast?**  
slow — с LLM-as-a-Judge (DeepEval), 5–10 мин. fast — без LLM, ~1 мин. По умолчанию: pytest -m "not slow".

**87. Как DeepEval интегрирован в тесты?**  
test_atomic.py; AnswerRelevancyMetric для slow; deepeval set-ollama. Тесты проверяют релевантность ответов Retriever, Command, Analyst.

**88. Почему нет интеграционных тестов с реальной БД?**  
Используется in-memory VectorMemory (conftest). Интеграционные с PostgreSQL требуют поднятия БД. Можно добавить: pytest с docker; или testcontainers.

**89. Как бы вы тестировали RouterAgent без LLM?**  
Mock LLMClient; подмена ответов (retrieval, command, analysis); проверка выбора агента. Или rule-based fallback при недоступности LLM.

**90. Почему нет тестов на load?**  
Не в scope MVP. Для production: locust, k6; нагрузочные тесты; профилирование.

**91. Как бы вы добавили regression-тесты для промптов?**  
Golden dataset: запрос → ожидаемый ответ/формат; проверка при изменении промптов; snapshot testing.

**92. Почему нет E2E-тестов для API?**

## Production и масштабирование (93–100)

**93. Как бы вы деплоили atomic в production?**  
Dockerfile для приложения; docker-compose с atomic, postgres, (опционально) Langfuse; или Kubernetes; env для конфигурации; CI/CD.

**94. Почему нет контейнеризации самого приложения?**  
Демо-проект; локальный запуск через uv run. Для production — Dockerfile.

**95. Как бы вы масштабировали API горизонтально?**  
Несколько инстансов за load balancer; stateless API; shared state (Redis, PostgreSQL) для ChatMemory; CodeExecutor — отдельный сервис с очередью.

**96. Почему нет явного graceful shutdown?**  
uvicorn поддерживает SIGTERM. Для production: обработка shutdown; завершение активных запросов; закрытие соединений.

**97. Как бы вы обрабатывали очереди запросов?**  
Celery, RQ, или Kafka; worker для async обработки; результат через WebSocket или polling. Для высокой нагрузки — обязательны.

**98. Почему нет явной стратегии кэширования?**  
Не в scope MVP. Для production: кэш ответов (Redis) по hash(query+context); кэш эмбеддингов; TTL.

**99. Как бы вы мониторили стоимость вызовов LLM?**  
Langfuse даёт стоимость по токенам; экспорт в Prometheus; дашборды; алерты при превышении лимитов.

**100. Какие ограничения проекта вы считаете критичными для production?**  
(1) CodeExecutor без sandbox — риск безопасности.  
(2) Нет аутентификации и rate limiting.  
(3) Один глобальный workflow — нет изоляции пользователей.  
(4) Нет chunking — ограничение качества RAG на больших документах.  
(5) Синхронный API — ограничение throughput.  
(6) Нет миграций БД — риск при изменениях схемы.
